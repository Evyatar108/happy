# Critique: `plans/async-events-design.md` (pre-write review)

## On the proposed direction

Your leaning — **app-owned watchers + MCP notification bridge / session subscription, mailbox preserved for intra-session** — is the right shape under the stated constraints. It correctly resists "exit-and-respawn" (blunt, loses turn state, false-wake prone) and avoids inventing a transport when the rmcp client already speaks bidi. Below are the issues I think you'd regret not surfacing in the doc.

## Blocking issues (must address in the doc)

**B1. The doc must commit to one of two architecturally distinct things and stop calling both "the bridge."**
There are two candidates being smeared together:
  - (d) **app-server → session push** (Socket.IO / app-server `bespoke_event_handling` plumbing): owned by happy-server, lives outside Codex, no upstream patch, but **only delivers when the agent is online to the app-server**.
  - (a) **MCP notification bridge** (`channels-research §6.1`): in-process, requires the two upstream seams in `logging_client_handler.rs` + `connection_manager.rs`, but is the only one that can fire when the agent is mid-session with no app-server attached.
  These are not interchangeable — they have different liveness, auth, and conflict-surface profiles. The doc should pick **MCP notification bridge as the in-session delivery path** and treat app-server push as a different layer (cross-session / cross-device fan-out), not as an alternative.
  *Severity: Blocking — without this split, candidate (d) keeps looking "simpler" than it is.*

**B2. None of the four use cases is naturally an MCP server today. The doc must name *who runs the MCP server* per use case, or the bridge is a transport with no producers.**
  - git events → needs a **codex-local "watcher MCP server"** (stdio, spawned by the session) that tails `git fsmonitor` / polls `refs/remotes/origin/main`. Not free — somebody writes it.
  - fs change under cwd → same shape; one watcher MCP server that exposes `notifications/fs/changed` (notify crate or `chokidar`).
  - periodic timers → trivial MCP server, but arguably overkill vs. a one-line in-core ticker emitting `EventMsg::Tick`.
  - inter-agent → already has the mailbox fast path; routing it through MCP is a regression in latency and complexity.
  Recommend the doc explicitly enumerate **producer ownership** per use case in a small table, and acknowledge that picking "MCP bridge" implies **also building one small watcher MCP server** (or an in-process `Service` impl over the in-memory duplex transport that `rmcp_client.rs` already supports — that's the cheap path and worth calling out).
  *Severity: Blocking.*

**B3. "Smallest viable subset that covers all four" is not the bridge alone.**
The honest minimum is:
  1. MCP notification bridge (channels-research §6.1) for in-session push — covers git/fs/timer if a watcher server exists.
  2. **Keep mailbox** for inter-agent (do not re-route through MCP — it's already preempt-safe at commentary boundaries via `turn.rs`).
  3. One in-process watcher `Service` (or stdio sidecar) that emits git + fs + timer notifications, exposed via the existing in-process duplex transport.
  No app-server subscription RPC needed for v1. The doc should explicitly **defer (d)** to "cross-device fan-out, post-v1."
  *Severity: Blocking — otherwise the design over-scopes and ships nothing.*

## Non-blocking issues (should address)

**N1. False-wake budget for "commit lands on main" is under-specified.**
`git fsmonitor` and naive `inotify`/`ReadDirectoryChangesW` on `.git/` will fire on *every* index/HEAD/lockfile churn — pre-commit hooks, `git status` from another shell, IDE indexers. The doc should mandate a **small predicate layer in the watcher server** (e.g. "fire only when `refs/heads/main` SHA changes from the last reported value") rather than push raw fs events to the agent and rely on the LLM to filter. This is the single largest false-wake source in the four use cases.
*Recommended fix:* watcher server stores last-seen ref SHA per watched ref; only emits on transition. Same pattern for `notifications/fs/changed` — debounce + path-glob filter server-side, never client-side.

**N2. Replay-after-restart is not free, and the doc should pick a stance.**
happy-server's `eventRouter.ts` has ordered push + replay buffer for Socket.IO clients — that's *cross-session* replay. The MCP bridge has **no replay**: if the agent restarts mid-turn (compact, OOM, `/clear`), every queued notification is lost because the rmcp connection is torn down. Two acceptable stances; pick one explicitly:
  (i) "v1 is best-effort, no replay; producers must be queryable so the agent can re-poll on resume" (cheap; matches MCP semantics).
  (ii) "watcher servers persist a monotonically-increasing seq per subscription and the agent passes last-seen seq on `initialize`" (more work, one-day add-on).
  Don't leave it implicit.

**N3. Backpressure / volume cap is correctly flagged in `channels-research §6.4 ¶3` but the doc should restate it as a hard requirement, not a "tunable."** A chatty fs watcher under a `node_modules/` write storm will trivially exceed `tx_event` capacity. Per-server token bucket + drop-with-counter, default conservative (e.g. 50/s), surfaced as a warning event so the agent knows it dropped.

**N4. Auth model for the watcher server.** stdio-spawned-by-session is the only safe v1 — inherits session trust, no network surface. The doc should explicitly **forbid HTTP/WS-transported watcher servers in v1** to avoid a "any localhost process can wake my agent" foot-gun. Streamable HTTP transport in `rmcp_client.rs` exists but should not be wired to watchers without an auth story.

**N5. Hooks (candidate e) are a red herring for these four use cases.** Hooks fire on *codex's own* lifecycle (PreToolUse, Stop, etc.) — none of the four use cases is a codex lifecycle event. Mention and dismiss in the candidate comparison; do not ship "hybrid using hooks" because it doesn't actually buy anything for git/fs/timer/inter-agent.

## Suggestions

**S1.** For the "is there a simpler decomposition?" question — yes: **one notification bridge + one watcher Service trait + mailbox**. That's three boxes, one upstream seam (the §6.1 patch), zero new transports. Lead the doc with this as the recommendation and present (b/c/d) as rejected alternatives with one-line reasons.

**S2.** Use the in-process duplex transport (already in `rmcp_client.rs`) for the v1 watcher rather than spawning a stdio sidecar. Lower latency, no process management, same `Service` trait surface — and it sidesteps N4 entirely.

**S3.** Explicitly carve out: **no hard mid-LLM-call preemption**. Match Claude Code channels' "between turns" semantics (per `channels-research §1.4`) and codex's existing mailbox preempt point in `turn.rs`. Pretending otherwise will invite scope creep into model-call cancellation, which is a different multi-week project.

**S4.** Add a short "what we're *not* solving in v1" section: cross-device fan-out, persistent replay across restarts, sampling-from-watcher, Claude-style permission relay. These are all reasonable v2 items and naming them prevents reviewers from re-litigating scope.

## Blind spots in your stated leaning

- You said "app-owned event watchers" — be careful that doesn't drift into "happy-app/happy-server owns the watcher and pushes via Socket.IO." That couples async events to the app-server being attached, which breaks headless `happy` CLI runs and TUI users. Watchers should be **codex-session-owned** (in-process MCP `Service`), with app-server fan-out as an *additional* sink, not the primary one.
- "MCP notification bridge" alone is a transport without producers — easy to forget that `channels-research` only covers the bridge half. The watcher-server half is unscoped work and should be acknowledged in the effort estimate.
- Mailbox preemption already does ~70% of what people imagine "async events" doing for the inter-agent case; the doc should make that explicit so reviewers don't ask "why not put inter-agent through the bridge too?"
