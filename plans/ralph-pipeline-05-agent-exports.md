## Context

Plans 01тАУ04 produce data for the React app. This plan turns the same data into a clean machine-readable interface for agents. The goal: an agent picking up cold should read ONE file (`plans/overview-snapshot.json`) and have everything it needs тАФ task list, pipeline state, recommendations, dependency graph, recent activity.

This plan also addresses the issue that `plans/overview-ralph-state.js` is JS-only (requires `new Function` eval). Agents should be able to read JSON directly.

## Snapshot schema

```ts
export interface Snapshot {
    generatedAt: string             // ISO
    generatedFromCommit: string     // short git SHA
    schemaVersion: number           // increment on breaking changes; v1 for this plan
    tasks: SnapshotTask[]           // merged OverviewTask + RalphPipelineState
    runs: RunRecord[]               // copy of OverviewData.runs[]
    recommendations: Recommendation[]  // from overview-recommendations.json (or empty if Plan 04 not shipped)
    dependencyGraph: DependencyGraph   // from overview-dependency-graph.json (or empty)
    runDurations: Record<string, number>  // runId тЖТ hours
    unmatched: Array<{ kind: string; slug: string; reason: string }>
    unmatchedSummary: Record<string, number>
}

export interface SnapshotTask extends OverviewTask {
    ralph?: RalphPipelineState     // pulled from ralphState.byTaskId[task.id]
}

export interface ActivityEvent {
    ts: string                      // ISO
    slug: string
    taskId?: string                 // resolved via matching at event time; absent if unmatched
    prevStage?: RalphStage          // null when first observed
    newStage?: RalphStage           // null when slug deleted
    changedFields: string[]
    reason: 'sync' | 'watch-event' | 'manual'
}
```

`SnapshotTask` extends `OverviewTask` rather than carrying `ralph` as a sibling so agents can iterate `snapshot.tasks` with everything attached.

## Activity log rotation

`plans/overview-activity.jsonl` grows monotonically. Cap at 1000 lines; on overflow, rotate:

- Move current file to `plans/overview-activity.1.jsonl` (single backup, overwritten).
- Start a fresh `plans/overview-activity.jsonl`.

This is best-effort тАФ if the process crashes mid-rotation, the worst case is a slightly-too-large primary file or a missing backup. Acceptable.

## Common mistakes / confusion points

1. **`overview-data.json` is generated, not hand-edited.** It's a derived twin. Bookkeepers continue editing `overview-data.js`. The JSON twin exists for agents that prefer not to eval JS.
2. **`tasks/INDEX.md` is generated.** Header comment makes this clear. Bookkeepers don't hand-edit it. The legacy `tasks/prd-*.md` files stay as historical artifacts but the canonical index is now this generated file.
3. **Activity events are derived inside `mergeAndWrite` and appended AFTER a successful merge.** The diff lives in `deriveActivityEvents` (scripts/lib/sync-core.mjs), which compares `currentState.byTaskId` against the new `state.byTaskId` while both halves are still in scope. The watcher (and one-shot CLI) appends `result.activityEvents` only after `mergeAndWrite` resolves successfully тАФ still inside the lock window тАФ so a failed sidecar write does not leak phantom events.
4. **Schema version bumps on breaking changes only.** Adding a new optional field is non-breaking. Renaming or removing a field is breaking.
5. **Activity rotation is best-effort.** Don't depend on the activity log for state тАФ use the snapshot. The activity log is for "what just changed?" debugging and the RecentActivity sidebar (Plan 07).
6. **`overview-data.json` is read-only.** The hand-curated source is `overview-data.js`. Tools that want to edit data should edit the `.js` file. (MCP `set_override` in Plan 09 writes the `.js` file via structured edit, not the `.json` twin.)
7. **Don't include Plan 07/08 fields prematurely.** Plan 07 adds `deferredQuestionsCount`; Plan 08 adds `crewSessions`. The snapshot type is extensible тАФ when those plans ship, they extend `RalphPipelineState` and the snapshot automatically includes the new fields. Don't pre-define them here.
