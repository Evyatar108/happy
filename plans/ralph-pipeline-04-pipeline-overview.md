## Scoring rubric

Default weights (override via `recommendations.weights` block in `.ralph/overview-config.json`):

| Input | Weight | Mapping |
|---|---|---|
| **Stage urgency** | 40 | `review-fix` = 1.0, `replan-pending` = 0.95, `plan-ready` = 0.9, `reviewing` = 0.7, `implementing` = 0.6, `blocked` = 0.5, `planning` = 0.4, `brainstorm-ready` = 0.3, `brainstorming` = 0.2, `shipped` = 0.0 |
| **Dependency state** | 30 | Unblocked = 1.0 (all `userStories[].dependencies[]` have `passes: true` in their referenced jobs), Partially blocked = 0.5, Fully blocked = 0.0 |
| **Freshness** | 20 | Linear decay from 1.0 (touched today) to 0.0 (touched â‰Ą14 days ago) based on `lastUpdatedAt` |
| **Priority** | 10 | `OverviewTask.priority` if numeric (normalized 0..1) else 0.5 |

`score = sum(weight_i * value_i) / sum(weight_i)`, range `[0, 1]`. Sort desc. Top 20 by default.

Each `Recommendation` carries `{ taskId, score, stage, reasons: string[] }` where `reasons` lists the top 2-3 contributing factors (e.g. `["review-fix stage", "unblocked", "not touched in 9 days"]`).

## Common mistakes / confusion points

1. **The histogram uses single-select; the toolbar group uses multi-select.** Both write to `filters.ralphStage`. The histogram clears and sets `new Set([stage])`; the toolbar toggles within the set. This is intentional â€” the histogram is for triage (pick one), the toolbar is for power filtering (pick several).
2. **`runDurations` is a transient home.** Plan 05 absorbs it into `overview-snapshot.json`. Don't build consumer code against `OverviewRalphState.runDurations` long-term â€” when Plan 05 ships, the field migrates.
3. **`derive-dependency-graph.mjs` reads `OverviewTask.blocks` but the field is bookkeeper-edited.** Don't auto-populate `blocks` from any source â€” it's an explicit signal from the bookkeeper.
4. **Don't bypass `score-recommendations.mjs` in the UI.** The histogram and `/triage` skill (Plan 06) and MCP server (Plan 09) all consume the precomputed file. The UI does NOT call `scoreRecommendations` at render time â€” the watcher / sync writes the file, the UI reads it.
5. **Empty state matters.** A user opening the dashboard for the first time (no `.ralph/` yet) should not see broken histogram chips â€” show the empty-state message instead.
