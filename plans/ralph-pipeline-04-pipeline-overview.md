## Context

Plan 03 gives every task a stage chip on its row. With ~18 tasks, scanning 18 chips to find "where are my bottlenecks?" takes ~10 seconds. A horizontal histogram of stage counts at the top of the dashboard answers the same question in ~0.5 seconds ÔÇö and doubles as a clickable filter UI. This plan adds:

1. `PipelineOverview.tsx` ÔÇö aggregate stage histogram rendered between `Toolbar` and `Kanban`.
2. `overview-recommendations.json` ÔÇö ranked "next task to work on" list generated by the sync.
3. `overview-dependency-graph.json` ÔÇö task DAG export for agents and a small inline DAG in the chip tooltip.
4. Cost/time annotations on `OverviewData.runs[]` for historical-duration analysis.

Together these turn the dashboard from "see current state" into "prioritize next action."

## Scoring rubric

Default weights (override via `recommendations.weights` block in `.ralph/overview-config.json`):

| Input | Weight | Mapping |
|---|---|---|
| **Stage urgency** | 40 | `review-fix` = 1.0, `replan-pending` = 0.95, `plan-ready` = 0.9, `reviewing` = 0.7, `implementing` = 0.6, `blocked` = 0.5, `planning` = 0.4, `brainstorm-ready` = 0.3, `brainstorming` = 0.2, `shipped` = 0.0 |
| **Dependency state** | 30 | Unblocked = 1.0 (all `userStories[].dependencies[]` have `passes: true` in their referenced jobs), Partially blocked = 0.5, Fully blocked = 0.0 |
| **Freshness** | 20 | Linear decay from 1.0 (touched today) to 0.0 (touched Ôëą14 days ago) based on `lastUpdatedAt` |
| **Priority** | 10 | `OverviewTask.priority` if numeric (normalized 0..1) else 0.5 |

`score = sum(weight_i * value_i) / sum(weight_i)`, range `[0, 1]`. Sort desc. Top 20 by default.

Each `Recommendation` carries `{ taskId, score, stage, reasons: string[] }` where `reasons` lists the top 2-3 contributing factors (e.g. `["review-fix stage", "unblocked", "not touched in 9 days"]`).

## Common mistakes / confusion points

1. **The histogram uses single-select; the toolbar group uses multi-select.** Both write to `filters.ralphStage`. The histogram clears and sets `new Set([stage])`; the toolbar toggles within the set. This is intentional ÔÇö the histogram is for triage (pick one), the toolbar is for power filtering (pick several).
2. **`runDurations` lives in the snapshot.** Do not build consumer code against a `runDurations` field on `OverviewRalphState`; Plan 05's `overview-snapshot.json` is the durable home.
3. **`derive-dependency-graph.mjs` reads `OverviewTask.blocks` but the field is bookkeeper-edited.** Don't auto-populate `blocks` from any source ÔÇö it's an explicit signal from the bookkeeper.
4. **Don't bypass `score-recommendations.mjs` in the UI.** The histogram and `/triage` skill (Plan 06) and MCP server (Plan 09) all consume the precomputed file. The UI does NOT call `scoreRecommendations` at render time ÔÇö the watcher / sync writes the file, the UI reads it.
5. **Empty state matters.** A user opening the dashboard for the first time (no `.ralph/` yet) should not see broken histogram chips ÔÇö show the empty-state message instead.
